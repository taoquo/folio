01 · Executive Summary

Executive Summary

Production agent quality improved in Q2 because the team stopped treating latency, cost, and failure recovery as separate concerns. The operating model now centers on one shared scorecard, and the result is a 37% drop in user-visible failures without slowing release velocity.

Key Takeaways

  • User-visible failure rate fell from 4.3% to 2.7% after introducing replay-safe tool calls and runbook-based retries.
  • Cost per successful run dropped 18% once prompt caching and retrieval batching were measured as one workflow instead of separate optimizations.
  • The next constraint is not model quality; it is operator throughput during multi-service incidents.

02 · Background

Background & Problem Statement

This chapter defines the baseline: what changed in traffic, what broke most often, and why the prior operating model no longer fit the workload.

Current State

By the end of Q2, the platform served 1.8M agent runs per week across support, research, and internal automation. Mean completion time held near 8.4 seconds, but high-variance tool chains created long-tail delays and most escalations clustered in three services.

The Core Problem

The core problem was not a single weak model. It was a fragmented control plane: prompts were versioned in one place, tool policies in another, and incident learning in a third, so every improvement arrived late and only partially.

We were shipping agent behavior faster than we were shipping the operating discipline required to trust it.

03 · Methodology

Methodology & Findings

We combined trace review, incident postmortems, and cohort analysis of tool paths to separate noise from repeatable failure modes.

Approach

Every production run was labeled by task family, tool fan-out, retry depth, and human takeover status. We then reviewed the top decile of failure clusters and mapped changes back to release windows, policy shifts, and queue behavior.

Key Findings

Finding 1 - Retry safety reduced cascading failures

Idempotent tool wrappers removed duplicate writes in 91% of previously unsafe retry paths. In the two highest-volume workflows, that change alone accounted for 0.8 percentage points of failure-rate improvement.

Finding 2 - Retry safety reduced cascading failures

The most effective latency reduction came from queue shaping and retrieval prefetching, not model swaps. Teams that cut queue contention saw the p95 completion time improve by 22% without touching model selection.

The biggest trust gain came when the system failed fast and told us what to do next, not when it tried one more invisible recovery step. - Support operations lead, 2026-04-18
04 · Conclusions

Conclusions & Recommendations

Q2 proved that agent quality is now an operations problem as much as a model problem. The recommendations below focus on that shift.

Core Conclusions

  1. Unifying traces, policies, and prompts under one review rhythm is now the highest-leverage management move.
  2. Run-level cost controls work best when attached to task cohorts rather than global model limits.
  3. Incident response must be optimized for cross-service diagnosis, not only per-service uptime.

Recommended Next Steps

Create one weekly operating review with product, platform, and support in the same room; ship queue-shaping defaults to all retrieval-heavy workloads; and require every new tool path to declare retry safety before launch.

Call to Action
Merge prompt, trace, and incident review into one operator dashboard before scaling the next agent workflow.
